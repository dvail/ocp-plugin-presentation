What is the Console Plugin?

ACS security data embedded directly in OpenShift Console

  • Security tabs on Deployments, StatefulSets, DaemonSets, CronJobs, Namespaces, Projects
  • Top-level /acs/security/vulnerabilities route
  • Image & CVE detail pages
  • No separate login — uses the user's existing OCP session

Why?

  • OCP users want security data in context
  • Viewing a Deployment? See its CVEs right there
  • No app-switching, no separate auth
  • Same code as the standalone UI — not a separate product

Webpack Module Federation

The technology behind dynamic plugins

The Requirements

Embedding our UI inside another React app at runtime requires solving three problems:

  • Shared dependencies — React, Redux, router, etc. must exist as a single instance in the browser. Two copies of React = broken context, broken hooks.
  • Exposed integration points — The host needs a way to discover and load our components (routes, tabs, cards) without bundling them at build time.
  • Cohesive UI — Our components render inside the console's layout, share its styling (PatternFly), and participate in its routing.

Webpack Module Federation solves all three.

Host & Remote

Host

The container app that loads remotes at runtime

OpenShift Console is the host

Remote

A separately-built bundle that exposes modules

Our plugin is the remote

At build time, the remote generates a remoteEntry.js manifest.
At runtime, the host fetches this manifest and dynamically loads only the chunks it needs.

Shared Dependencies at Build Time

The host declares a list of shared modules. When webpack builds a remote:

  • Any import of a shared module becomes an async external reference instead of being bundled into the chunk
  • Our plugin's bundle literally does not contain React, Redux, react-router, etc.

This is the fundamental constraint: our code is compiled against one version of these libraries but runs against whatever version the console ships. Version mismatches can cause silent breakage.

How Console Discovers Plugins


apiVersion: console.openshift.io/v1
kind: ConsolePlugin
metadata:
  name: advanced-cluster-security
spec:
  displayName: Red Hat Advanced Cluster Security
  backend:
    type: Service
    service:
      name: sensor-proxy
      namespace: stackrox
      port: 443
      basePath: /proxy/central/static/ocp-plugin

Console reads this CRD → fetches manifest from the service → loads our bundles

Our Webpack Config

We don't configure ModuleFederationPlugin directly.
The SDK provides ConsoleRemotePlugin which wraps it. 


// webpack.ocp-plugin.config.js

        { type: 'console.context-provider', ... },

        // ...
    ],
})

Sharing Code Between Modes

Plugin entry points are thin wrappers around large shared components.
e.g. DeploymentPageVulnerabilities — the full vulnerability table with filtering, sorting, pagination, column management, and CVE detail links.

The wrapper customizes behavior through three patterns:

  • Props — e.g. showVulnerabilityStateTabs={false} to hide standalone-only UI
  • React Context — e.g. WorkloadCveViewContext provides configuration that shared components read via useContext, such as building URLs for intra-app links
  • Composition — wrapping shared components with plugin-specific chrome like NamespaceBar, loading states, or error boundaries

Example: Workload Security Tab


// ConsolePlugin/WorkloadSecurityTab/WorkloadSecurityTab.tsx
export function WorkloadSecurityTab() {
    useAnalyticsPageView();

    const context = useDefaultWorkloadCveViewContext(); // ← context DI
    const pagination = useURLPagination(DEFAULT_VM_PAGE_SIZE);
    const { searchFilter, setSearchFilter } = useURLSearch();
    const { ns, name } = useParams();
    const { id, isLoading, error } = useWorkloadId({ ns, name });

    return (
        // Context provides config that shared components consume
        <WorkloadCveViewContext.Provider value={context}>
            {isLoading && <Spinner />}
            {error && <EmptyStateTemplate ... />}
            {id && (
                // Large shared component — full vuln table w/ filtering,
                // sorting, pagination, column management
                <DeploymentPageVulnerabilities
                    deploymentId={id}
                    pagination={pagination}
                    showVulnerabilityStateTabs={false} // ← prop DI
                    vulnerabilityState="OBSERVED"
                    searchFilter={searchFilter}
                    setSearchFilter={setSearchFilter}
                />
            )}
        </WorkloadCveViewContext.Provider>
    );
}

consoleFetchAxiosAdapter.ts overrides axios.defaults.adapter to route requests through the console's consoleFetch.

Shared services call axios identically in both modes.

Authentication, RBAC & Request Flow

No separate login required

Step 1: Component makes a request

sequenceDiagram
    participant C as Plugin Component
    participant A as Axios
    C->>A: axios.get('/v1/deployments')
    Note over A: Standard axios call, same code as standalone

The component doesn't know it's running inside the console.
It uses the same axios instance as the standalone app.

Step 2: Adapter intercepts & adds auth

sequenceDiagram
    participant A as Axios
    participant AD as consoleFetchAxiosAdapter
    participant CF as consoleFetch (SDK)
    A->>AD: adapter(config)
    Note over AD: Injects ACS-AUTH-NAMESPACE-SCOPE header from active namespace
    AD->>CF: consoleFetch(url, options)
    Note over CF: SDK adds auth cookies + X-CSRFToken

PluginProvider overrides axios.defaults.adapter
so all requests go through consoleFetch instead of XMLHttpRequest.

Step 3: Console proxies to sensor-proxy

sequenceDiagram
    participant CF as consoleFetch
    participant CP as Console Proxy
    participant SP as sensor-proxy
    CF->>CP: /api/proxy/plugin/advanced-cluster-security/api-service/...
    Note over CP: Console backend resolves user session to bearer token
    CP->>SP: Request + Authorization: Bearer ...
    Note over SP: Receives the user's OCP bearer token

Console auth uses cookies in the browser. The console backend resolves the session and adds a bearer token on the backend-to-backend proxy hop.
In dev: the token from oc login --web. In prod: the token from the user's authenticated OpenShift session.

Step 4: sensor-proxy RBAC

sequenceDiagram
    participant SP as sensor-proxy
    participant K8s as Kubernetes API
    participant C as Central

    SP->>K8s: TokenReview (validate bearer token)
    K8s-->>SP: User identity (username, groups, UID)

    SP->>K8s: SubjectAccessReview
    Note over SP,K8s: Can user get/list deployments, pods, statefulsets, etc. in requested namespace?
    K8s-->>SP: Allowed / Denied

    SP->>C: Request + dynamic access scope
    Note over C: Read-only Deployment + Image permissions, scoped to namespace

sensor-proxy RBAC Model

  • Authentication: validates the OCP bearer token via Kubernetes TokenReview (cached for 3 min)
  • Authorization: issues a SubjectAccessReview checking get + list on deployment-like resources (Deployments, StatefulSets, Pods, CronJobs, etc.) in the requested namespace
  • Scoping: creates a dynamic access scope on the forwarded request — either full-cluster or namespace-restricted
  • Least privilege: tokens issued to Central are restricted to read-only Deployment + Image permissions only

Namespace scope header: ACS-AUTH-NAMESPACE-SCOPE = specific namespace, * for all, or absent for endpoints that don't need scoping. The SSAR check and dynamic scope both key off this header.

Full Picture

sequenceDiagram
    participant P as Plugin
    participant AX as Adapter
    participant CF as consoleFetch
    participant CP as Proxy
    participant SP as sensor-proxy
    participant K as K8s
    participant C as Central

    P->>AX: axios.get('/v1/...')
    AX->>AX: Add scope header
    AX->>CF: consoleFetch(url)
    CF->>CF: Add cookies + CSRF
    CF->>CP: /api/proxy/plugin/...
    CP->>SP: Forward + auth
    SP->>K: TokenReview + SSAR
    K-->>SP: Identity + authz
    SP->>C: Dynamic access scope
    C-->>SP: Filtered data
    SP-->>P: Response

Adding Plugin Features

Decision Tree

Modifying an existing plugin view?
→ Edit shared code in Containers/. It works in both modes.

Adding a new extension?
→ Three steps:

  1. Create entry point in ConsolePlugin/
  2. Register in exposedModules
  3. Add console extension(s)

Example: Security Overview Dashboard Tab

Let's add a Security tab to the main cluster overview dashboard using console.dashboards/tab + console.dashboards/card.

Step 1: Create the Card Component


// src/ConsolePlugin/OverviewSecurityCard/OverviewSecurityCard.tsx

import { Card, CardHeader, CardTitle, CardBody } from '@patternfly/react-core';

export function OverviewSecurityCard() {
    return (
        <Card>
            <CardHeader>
                <CardTitle>Security Overview</CardTitle>
            </CardHeader>
            <CardBody>
                Hello from ACS!
            </CardBody>
        </Card>
    );
}

Step 2: Register the Module


// webpack.ocp-plugin.config.js

exposedModules: {
    context: './ConsolePlugin/PluginProvider',
    // ... existing modules ...

    // New dashboard card
    OverviewSecurityCard: './ConsolePlugin/OverviewSecurityCard/OverviewSecurityCard',
},

Step 3: Add Console Extensions


// webpack.ocp-plugin.config.js — extensions array

// Register a new dashboard tab
{
    type: 'console.dashboards/tab',
    properties: {
        id: 'acs-security-overview',
        navSection: 'home',
        title: 'Security',
    },
},

// Add a card to that tab
{
    type: 'console.dashboards/card',
    properties: {
        tab: 'acs-security-overview',
        position: 'LEFT',
        component: { $codeRef: 'OverviewSecurityCard.OverviewSecurityCard' },
    },
},

Compatibility & Long-Term Support

The console controls our runtime

Shared Singleton Modules

Console provides these as singletons. Our bundles don't contain them —
the console's version is what runs, regardless of our package.json.

Module Console 4.19 Our package.json
react ^17.0.1 ^18.2.0 ⚠
react-router-dom 5.3.x ^5.3.4 ✓
react-redux 7.2.9 ^7.2.6 ✓
redux 4.0.1 ^4.1.2 ✓

React 17 at Runtime

Our package.json says React 18, but console 4.19 ships React 17.
The plugin runs on React 17 at runtime.

Avoid in plugin code paths:

  • useId(), useDeferredValue(), useTransition()
  • createRoot() / hydrateRoot()
  • Relying on automatic batching behavior

Code that runs only in the standalone UI can use React 18 features freely.

PatternFly: The CSS Trap

@patternfly/react-core is not a shared singleton — we bundle the JS ourselves. So why does PF version matter?

Because of CSS.

  • ConsoleRemotePlugin aliases all PatternFly CSS to false at build time — plugins cannot bundle their own PF stylesheets
  • Console loads PatternFly CSS globally for all plugins via vendor.scss
  • PF5 uses .pf-c-* class names, PF6 uses .pf-v6-c-*
  • If our components emit PF5 class names but the console only loads PF6 CSS → no matching styles → visual breakage

PatternFly Support Timeline

Console Version PF CSS Loaded Status
4.19 – 4.21 PF6 + PF5 (transitional) Both class names styled
4.22+ PF6 only PF5 classes unstyled

Our app is already fully migrated to PF6 — we hit this issue during the tech preview on PF5, which is why we had to disable the plugin on OCP ≥4.22. Eventually the console will move to PF7, and we'll need to plan a migration strategy then.

What's Coming: Console main

Module 4.19 (current) main (future) Impact on us
react ^17.0.1 ^18.3.1 Aligns with us
redux 4.0.1 ~5.0.1 No impact*
react-redux 7.2.9 ~9.2.0 No impact*
react-router 5.3.x ~7.13.1 See next slide
react-i18next ^11.12.0 ~16.5.8 Not used by plugin

* Redux and react-redux are not used in any plugin code paths — all shared components use React Context + Apollo Client for state management.

react-router Migration Path

Console main has moved to react-router v7.

  • The react-router-dom-v5-compat shared module is aliased to react-router v7 in the latest console
  • This compat layer is expected to remain available through approximately OCP 5.2
  • Our plugin currently uses react-router-dom-v5-compat for v6-style APIs (useParams, useNavigate)
  • These v6 APIs are forward-compatible with v7, so the migration risk is low

Action: Continue using react-router-dom-v5-compat imports. When the compat shim is removed (~5.2), rename imports to react-router.

Compatibility Strategy

1. Monitor the source of truth
shared-modules-meta.ts and the SDK README on each release branch.

2. SDK version pinning
Our SDK is pinned to 4.19.0. Upgrading requires testing against the target console version.

3. Minimum API version
dependencies: { '@console/pluginAPI': '>=4.19.0' }
Console won't load the plugin if it doesn't satisfy this constraint.

Future Discussions: Planning Ahead

  • Planning core dependency upgrades (react-router, PatternFly) around the console's release timeline — when do we start, how do we phase it?
  • sensor-proxy ↔ Central version skew — how do we test and guarantee compatibility across releases?
  • New SDK extension types we should adopt?

Key Takeaways

  • Plugin code is ~2% wrapper, ~98% shared with standalone
  • All traffic goes through sensor-proxy — SSAR + dynamic access scopes + read-only tokens
  • Console controls shared deps at runtime — our bundles don't contain them
  • PatternFly CSS is the biggest compat risk — console blocks plugin PF CSS, loads its own globally
  • Adding features = thin wrapper + webpack config entries
